🔀 两条路径选择

A

单体仓库 · 全栈合一

把 nihaisha 资料 copy 到前端项目里，一个仓库管全部。数据直接 import 进 React，无需网络请求。

Pros

  • 结构简单，一个 npm install + dev 就能跑
  • 数据 import 零延迟
  • 不需要部署后端
  • ModelScope Static HTML 直接部署

Cons

  • 仓库变大（78MB 截图 + 源码）
  • 资料更新需要重新 build
  • 数据和前端紧耦合
  • npm build 时截图跟着打包
B

前后分离 · API 驱动

nihaisha 作为数据源，前端通过 API 拉取结构化数据。截图走 CDN/静态文件服务。

Pros

  • 职责分明，资料库和前端独立演进
  • nihaisha 更新，前端自动同步
  • 截图不参与 npm build，保持轻量
  • 后期可接入多端（Web + 小程序 + App）

Cons

  • 需要一个简单的后端服务
  • 需要额外部署一个 API server
  • 离线体验受影响
  • 初始工作量更大
💡 我的建议

分两步走：现在用方案 A（单体）快速交付——数据静态 import + 截图用路径索引不打包原图。 等 8 个必做功能做完、学习流跑通后，再切到方案 B——写一个轻量 Python/Node 脚本把 Markdown 转成 JSON API， 加一个 Express/FastAPI 做数据 + 截图服务，前端 fetch。这样不耽误当前进度。
