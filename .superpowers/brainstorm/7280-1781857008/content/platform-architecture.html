❓ 两个关键决策需要你定

Q1

数据格式：Markdown 还是结构化 JSON/YAML？

现在 nihaisha 是 Markdown 表格+列表。构建管道可以维持 MD 格式（易编辑）自动解析，也可以要求数据源提供结构化的 JSON/YAML（易查询）。

我的建议：保持 Markdown 为主（人写舒服），构建时转成 JSON 索引。关键结构化数据（方剂参数、穴位坐标）用 YAML frontmatter。

Q2

API 层用什么语言？

两个选择：

Node.js/Express

  • 和前端同一个语言栈
  • 可以写 TS 共享类型
  • Vite SSR/中间件可以复用

Python/FastAPI

  • nihaisha 的脚本已经是 Python
  • Markdown 解析生态好
  • 后续 AI/LLM 集成方便

听你的。两种都能做，差别不大。

Q3

仓库组织方式？

三个独立仓库，还是用 Monorepo（一个仓库，packages/ 下分 data / api / web）？

我的建议：三个独立仓库，通过 git submodule 引用数据层。因为 nihaisha 本身是独立项目，不想被合并。Monorepo 等你上手后用 turborepo 切也不晚。
